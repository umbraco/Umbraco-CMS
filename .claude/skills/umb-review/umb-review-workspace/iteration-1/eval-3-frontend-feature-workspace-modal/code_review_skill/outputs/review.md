# Code Review: PR #22268 -- Current User Workspace Modal

**PR**: #22268 (29 files, +960/-8)
**Branch**: `v17/feature/current-user-workspace` targeting `origin/main`
**Scope**: Full-stack feature adding a current user workspace modal, allowing users who lack Users section access to edit their own profile settings (language) and avatar.

---

## Summary

This PR introduces a "current user workspace" -- a sidebar modal that lets logged-in backoffice users edit their own profile settings (currently language preference) and manage their avatar, without needing access to the Users section. The PR spans:

- **Backend**: New `ClearAvatarCurrentUserController`, `UpdateCurrentUserProfileController`, `CurrentUserUpdateModel`, `IUserService.UpdateCurrentUserAsync`, and supporting factory/view model code.
- **Frontend**: New `SectionUserNoPermission` condition, workspace modal elements (avatar, profile settings), action to open the modal, and repository/data-source extensions.
- **Tests**: Two integration tests for the new `UpdateCurrentUserAsync` service method.

## Findings

### Critical

#### 1. Breaking change: new method on public `IUserService` interface without default implementation

**File**: `src/Umbraco.Core/Services/IUserService.cs`

`IUserService` is a public interface in `Umbraco.Core`. Adding `UpdateCurrentUserAsync` as an abstract member (no default implementation) is a **binary breaking change** for any external consumer that implements `IUserService`. The repository's CLAUDE.md explicitly documents the "Default Interface Implementation" pattern for exactly this scenario (Section 5.3).

The new method should have a default implementation, for example:

```csharp
Task<Attempt<IUser?, UserOperationStatus>> UpdateCurrentUserAsync(CurrentUserUpdateModel model)
    => Task.FromResult(Attempt.FailWithStatus<IUser?, UserOperationStatus>(
        UserOperationStatus.Unknown, default));
```

Or delegate to `UpdateAsync` if a reasonable mapping exists.

#### 2. Hand-edited auto-generated files (`sdk.gen.ts`, `types.gen.ts`)

**Files**: `src/Umbraco.Web.UI.Client/src/packages/core/backend-api/sdk.gen.ts`, `types.gen.ts`

Both files carry the header `// This file is auto-generated by @hey-api/openapi-ts`. The PR hand-edits them to add `deleteUserCurrentAvatar`, `putUserCurrentProfile`, and their types. This is problematic because:

- The next OpenAPI code generation run will overwrite these hand-edits.
- The hand-edited SDK methods reuse response/error types from unrelated endpoints (`DeleteUserAvatarByIdResponses`/`DeleteUserAvatarByIdErrors` for the current-user avatar delete, and `PutUserByIdResponses`/`PutUserByIdErrors` for the profile update). This works today because the response shapes are identical, but it creates a fragile coupling -- changing the response of the "by ID" endpoints would break the "current" endpoints silently.

The correct approach is to regenerate these files from the updated OpenAPI specification after the backend controllers are merged.

### Major

#### 3. Modal save always submits regardless of API errors

**File**: `src/Umbraco.Web.UI.Client/src/packages/user/current-user/modals/current-user-workspace/current-user-workspace-modal.element.ts` (lines ~805-808 in the diff)

```typescript
private async _save() {
    await this._profileSettings?.save();
    this.modalContext?.submit();
}
```

`save()` on the profile settings element calls `this.#currentUserRepository.updateProfile(...)` but does not propagate the error. Even if the API call fails, the modal submits as if the save succeeded. The user receives no feedback that their changes were not persisted.

The `save()` method should return a success/failure indicator, and `_save()` should only call `submit()` on success.

#### 4. `ClearAvatarCurrentUserController.ClearAvatar` declares but ignores `cancellationToken`

**File**: `src/Umbraco.Cms.Api.Management/Controllers/User/Current/ClearAvatarCurrentUserController.cs` (line 58)

The method signature includes `CancellationToken cancellationToken` but never passes it downstream to `_userService.ClearAvatarAsync(userKey)` or `_authorizationService.AuthorizeResourceAsync(...)`. While the parameter satisfies ASP.NET Core's expectation for cancelable endpoints, not passing it to the service means the operation cannot be canceled once started. Either pass it through or remove the parameter to avoid confusion.

### Minor

#### 5. Typo in test: variable named `modal` instead of `model`

**File**: `tests/Umbraco.Tests.Integration/Umbraco.Core/Services/UserServiceCrudTests.Update.cs` (line ~1225 in diff)

```csharp
var modal = new CurrentUserUpdateModel
{
    ExistingUserKey = createdUser.Key,
    LanguageIsoCode = "da"
};

var result = await userService.UpdateCurrentUserAsync(modal);
```

The variable is named `modal` (referencing UI concept) instead of `model`. The second test correctly uses `model`. This should be `model` for consistency.

#### 6. No input validation on `UpdateCurrentUserRequestModel.LanguageIsoCode`

**File**: `src/Umbraco.Cms.Api.Management/ViewModels/User/UpdateCurrentUserRequestModel.cs`

`LanguageIsoCode` defaults to `string.Empty` and has no validation attributes. An empty string or arbitrary invalid ISO code can be submitted and will be persisted to the database without validation. The existing `UpdateUserRequestModel` has the same issue (no attribute-based validation), so this is consistent with the current pattern, but worth noting as the service layer (`MapCurrentUserUpdate`) blindly assigns `target.Language = source.LanguageIsoCode`.

#### 7. Trailing whitespace in `UserService.cs`

**File**: `src/Umbraco.Core/Services/UserService.cs` (lines ~2516-2517 in diff)

The diff adds two blank lines before `#endregion`:

```csharp
+
+
    #endregion
```

This appears to be accidental trailing whitespace that may trigger a formatting check failure.

#### 8. Hardcoded English notification strings in repository

**File**: `src/Umbraco.Web.UI.Client/src/packages/user/current-user/repository/current-user.repository.ts` (lines ~1115, ~1133 in diff)

```typescript
const notification = { data: { message: `Avatar uploaded` } };
// ...
const notification = { data: { message: `Avatar deleted` } };
```

These are hardcoded English strings rather than localized via `this.localize.term(...)`. The existing codebase has the same issue for `Password changed` (line 139 of the existing file), so this is a pre-existing pattern. However, since this PR introduces new user-facing messages, it would be a good opportunity to use localization keys instead.

## Positive Observations

- **Well-structured separation of concerns**: The feature cleanly separates avatar management, profile settings, and modal orchestration into distinct custom elements.
- **Follows existing patterns**: The `SectionUserNoPermissionCondition` mirrors the existing `SectionUserPermissionCondition` closely, with proper negation logic. The repository, data source, and condition all follow established codebase conventions.
- **Good test coverage**: The two integration tests verify both the happy path and the immutability of non-language fields, which is an important correctness guarantee.
- **Authorization correctly applied**: Both backend controllers check `UserPermissionByResource` policy before performing operations, consistent with the controller-per-operation authorization pattern.
- **Condition-based visibility**: Using `UMB_SECTION_USER_NO_PERMISSION_CONDITION_ALIAS` to conditionally show the workspace edit button only for users without Users section access is a clean approach that avoids duplicating edit UI.
- **Indentation fix**: The PR opportunistically corrects misaligned XML doc comments in `IUserPresentationFactory.cs`.
