# Code Review: PR #22268 -- Current User Workspace Modal

## Summary

This PR adds a "current user workspace" modal that allows users who do NOT have access to the Users section to edit their own profile (currently limited to changing their UI language) and manage their avatar. It introduces:

- **Backend**: Two new API controllers (`ClearAvatarCurrentUserController`, `UpdateCurrentUserProfileController`), a new service method on `IUserService`, a new domain model, presentation factory method, and request model.
- **Frontend**: A new workspace modal with avatar management and profile settings elements, a new `SectionUserNoPermission` condition, repository/data-source extensions, and generated API client additions.
- **Tests**: Two integration tests for `UpdateCurrentUserAsync`.

---

## Critical Issues

### 1. Breaking Change: New method on `IUserService` without default implementation
**Severity: Critical**
**Files**: `src/Umbraco.Core/Services/IUserService.cs` (diff line 274)

The new method `UpdateCurrentUserAsync(CurrentUserUpdateModel model)` is added directly to the `IUserService` interface without a default implementation. `IUserService` is a public interface in `Umbraco.Core` that external consumers (packages, custom implementations) may implement. Adding a member without a default implementation is a **binary breaking change**.

Per the repository's breaking change policy (CLAUDE.md section 5.3), new interface methods must provide a default implementation:

```csharp
Task<Attempt<IUser?, UserOperationStatus>> UpdateCurrentUserAsync(CurrentUserUpdateModel model)
    => throw new NotImplementedException();
// TODO (V19): Remove the default implementation.
```

Alternatively, consider using `UpdateAsync` with the performing user key set to the current user's own key, avoiding a new interface method altogether.

### 2. Missing language ISO code validation in `UpdateCurrentUserAsync`
**Severity: Critical**
**Files**: `src/Umbraco.Core/Services/UserService.cs` (diff lines 288-311)

The existing `UpdateAsync` method validates the language ISO code via `_isoCodeValidator.IsValid(model.LanguageIsoCode)` (line 1040 of the current file). The new `UpdateCurrentUserAsync` performs no validation at all -- it directly maps `source.LanguageIsoCode` to `target.Language` without checking whether the ISO code is valid. This allows setting an arbitrary or invalid string as the user's language, which could cause runtime errors or unexpected behavior in localization.

The method should validate:
```csharp
if (_isoCodeValidator.IsValid(model.LanguageIsoCode) is false)
{
    return Attempt.FailWithStatus<IUser?, UserOperationStatus>(UserOperationStatus.InvalidIsoCode, existingUser);
}
```

## High Severity Issues

### 3. `sdk.gen.ts` appears to be hand-edited with incorrect response types
**Severity: High**
**Files**: `src/Umbraco.Web.UI.Client/src/packages/core/backend-api/sdk.gen.ts` (diff lines 366-401)

The `sdk.gen.ts` file is auto-generated (`// This file is auto-generated by @hey-api/openapi-ts`). The new methods `deleteUserCurrentAvatar` and `putUserCurrentProfile` reference response/error types from the existing non-current-user endpoints:
- `deleteUserCurrentAvatar` uses `DeleteUserAvatarByIdResponses` and `DeleteUserAvatarByIdErrors`
- `putUserCurrentProfile` uses `PutUserByIdResponses` and `PutUserByIdErrors`

These are the types for `DELETE /user/avatar/{id}` and `PUT /user/{id}` respectively -- not the newly added current-user endpoints. If these endpoints have different error scenarios (the current-user endpoints don't have 403 unauthorized in the same way, for example), the types will be wrong.

This file should be regenerated from the OpenAPI spec after updating `OpenApi.json`, not hand-edited. Hand-editing generated files creates maintenance problems and risks being overwritten on the next generation.

### 4. `types.gen.ts` also appears to be hand-edited
**Severity: High**
**Files**: `src/Umbraco.Web.UI.Client/src/packages/core/backend-api/types.gen.ts` (diff lines 414-472)

Same concern as above. The new types (`UpdateCurrentUserRequestModel`, `DeleteUserCurrentAvatarData`, `PutUserCurrentProfileData`, etc.) appear hand-written in a generated file. While the types themselves look reasonable, they should be generated from the updated OpenAPI spec.

### 5. `IUserPresentationFactory` new method without default implementation
**Severity: High**
**Files**: `src/Umbraco.Cms.Api.Management/Factories/IUserPresentationFactory.cs` (diff line 178)

`CreateUpdateCurrentUserModelAsync` is added to the `IUserPresentationFactory` public interface without a default implementation. While this interface is in the Management API project (less likely to be implemented externally than `IUserService`), it is still a public interface and adding members without defaults is a breaking change. A default implementation should be provided.

## Medium Severity Issues

### 6. Missing `CancellationToken` forwarding in `ClearAvatarCurrentUserController`
**Severity: Medium**
**Files**: `src/Umbraco.Cms.Api.Management/Controllers/User/Current/ClearAvatarCurrentUserController.cs` (diff line 58)

The `ClearAvatar` action accepts a `CancellationToken cancellationToken` parameter but never uses it. While the existing `SetAvatarCurrentUserController` has the same pattern (accepts but doesn't forward), the `UpdateCurrentUserProfileController` doesn't accept one at all. Be consistent: either forward the token to service calls or remove the parameter.

### 7. No error handling or user feedback for failed save in the modal
**Severity: Medium**
**Files**: `current-user-workspace-modal.element.ts` (diff lines 805-808)

The `_save` method calls `await this._profileSettings?.save()` and then immediately calls `this.modalContext?.submit()`. If the save fails (e.g., server returns an error), the modal will still close as if the save succeeded. The `save()` method on the profile settings element calls `this.#currentUserRepository.updateProfile(...)` which returns `{ error }`, but that error is never checked.

```ts
private async _save() {
    await this._profileSettings?.save();
    this.modalContext?.submit(); // Always submits, even on error
}
```

The save should check for errors and only submit on success, or display an error notification.

### 8. Notification messages are not localized
**Severity: Medium**
**Files**: `current-user.repository.ts` (diff lines 1115-1116, 1133-1134)

The notification messages `"Avatar uploaded"` and `"Avatar deleted"` are hardcoded English strings. The existing `changePassword` method in the same repository also has this issue (`"Password changed"`), but new code should use localization:

```ts
const notification = { data: { message: this.localize.term('user_avatarUploaded') } };
```

Note: `UmbRepositoryBase` may not have `localize` available. If not, the notifications should use localization keys that the notification system can resolve.

### 9. Deep relative import across package boundary
**Severity: Medium**
**Files**: `current-user-workspace-avatar.element.ts` (diff line 622)

```ts
import '../../../../user/user/components/user-avatar/user-avatar.element.js';
```

This import traverses from `current-user/modals/current-user-workspace/` up to the `user/user/components/` directory. While technically within the same parent `user` package, this creates tight coupling across sub-packages. The `umb-user-avatar` element is registered as a side-effect import in `@umbraco-cms/backoffice/user` components. Consider importing from the package export path instead, or using `@umbraco-cms/backoffice/user` if the component is available there.

## Observations (Non-blocking)

### Architectural note on the "workspace" naming
The feature is named "current-user-workspace" but is implemented as a modal, not a true workspace (workspaces in Umbraco are typically full-page views with their own routing). The modal token uses `type: 'sidebar'`, which is appropriate for a lightweight editing experience. The naming could be confusing for developers familiar with the Umbraco workspace concept. Consider renaming to `current-user-profile-modal` or similar to avoid ambiguity.

### Authorization pattern is consistent
Both new controllers correctly follow the established pattern of authorizing via `IAuthorizationService.AuthorizeResourceAsync` with `UserPermissionResource.WithKeys(userKey)` and `AuthorizationPolicies.UserPermissionByResource`, matching the existing `SetAvatarCurrentUserController` and `GetCurrentUserController`.

### The `SectionUserNoPermissionCondition` is well-designed
The condition correctly inverts the logic of `SectionUserPermissionCondition` by using `!allowedSections.includes(this.config.match)`. This enables showing the edit button only for users who cannot access the Users section (where they would normally edit their profile through the full user workspace).

---

## Verdict: **Request Changes**

The two critical issues must be addressed before merging:

1. **Breaking change** on `IUserService.UpdateCurrentUserAsync` -- must add a default interface implementation or reconsider the approach.
2. **Missing ISO code validation** -- the new service method bypasses validation that exists in the parallel `UpdateAsync` method, creating a security/data-integrity gap.

The high-severity issues around hand-edited generated files should also be resolved to avoid maintenance headaches and type mismatches.
